A. Локальная демо-БД (Liquibase v4 + v5) — логины и пароли
================================================================================

После применения миграций в PostgreSQL создаются пользователи:

  Роль      Email              Пароль
  --------  -----------------  --------
  ADMIN     admin@kurash.uz    admin123
  COACH     coach@kurash.uz    coach123
  ATHLETE   athlete@kurash.uz  athlete123

У спортсмена athlete@kurash.uz в базе уже есть: тренировки, тесты ОФП, достижения,
черновик рекомендаций (сид v5), привязка к тренеру coach@kurash.uz.

Если вы регистрировали СВОЙ email через форму — готовых «демо»-строк в БД нет:
страницы будут пустыми, пока сами не занесёте тренировки, достижения, не нажмёте
«Сгенерировать» в рекомендациях и т.д.

Фронтенд (Vite, часто порт 5174) должен ходить на API бэкенда:
  VITE_API_BASE_URL=http://localhost:8080/api
Скопируйте frontend/.env.example → frontend/.env и при необходимости поправьте URL.


================================================================================
B. Зачем система и кто главные участники
================================================================================

Идея: единое место, где СПОРТСМЕН ведёт учёт подготовки и результатов, ТРЕНЕР
видит свою группу, АДМИНИСТРАТОР управляет учётными записями и привязками.

  • Спортсмен — основной «носитель» данных: его профиль (athletes), тренировки,
    тесты, достижения, заявки, рекомендации, медиа к достижениям — всё завязано
    на athlete_id.

  • Тренер — не «владелец» данных спортсмена, а связанный наставник: в БД у
    спортсмена поле coach_id → на запись в coaches. Пока админ не назначил тренера,
    тренер этого спортсмена в отчётах не увидит.

  • Администратор — управляет пользователями и ролями, назначает тренера
    спортсмену, может верифицировать достижения и т.д. (см. эндпоинты админки).


================================================================================
C. Бизнес-сущности и связи (упрощённая карта)
================================================================================

users
  └─ один пользователь = один логин (email, role: ADMIN | COACH | ATHLETE).

  Для ATHLETE обязательно есть профиль athletes (регион, вес, дата рождения, …)
  и опционально coach_id → coaches.

  Для COACH есть профиль coaches (регион, опыт, …), связанный с users.

Данные «по спортсмену» (все привязаны к athletes.id):

  trainings              — журнал тренировок (тип, длительность, интенсивность …)
  fitness_tests          — контрольные нормативы (подтягивания, бег …)
  achievements           — старт/результаты соревнований (уровень, медаль …)
  achievement_training_weeks — расписание недель под конкретное достижение (JSON)
  achievement_media      — фото/видео, прикреплённые к достижению (файлы на диске)
  recommendations        — сохранённые «рекомендации» (план недели, дефициты …)
  training_requests      — заявки спортсмена (например, пожелание по нагрузке)

Справочник / эталон:

  gold_standards           — средние «целевые» показатели по весовой категории;
                             используется модулем рекомендаций для сравнения
                             (если для категории спортсмена нет строки — часть
                             экранов может быть пустой или с подсказкой).

Внешняя интеграция (опционально):

  OpenAI (настройки в application.yml: app.ai.*) — для «инсайта» и черновика
  плана по журналу достижений (achievement insight). Без ключа API эвристика
  может работать частично или с заглушками — смотрите статус AI на экране журнала.


================================================================================
D. Кто что делает — типовые сценарии (бизнес-потоки)
================================================================================

D1. Новый спортсмен (регистрация ATHLETE)
  1) Регистрация → создаётся users + athletes.
  2) Входит в систему → JWT, доступ к разделам спортсмена.
  3) Заполняет профиль (ФИО, телефон, весовая, цели …).
  4) По мере жизни: добавляет тренировки, может создать заявку тренеру,
     смотрит рекомендации (сравнение с эталоном + генерация плана),
     вносит достижения и недели расписания в журнале, прикрепляет фото/видео.

D2. Привязка к тренеру (обязателен админ или уже заданный coach при регистрации)
  1) Админ в /admin выбирает спортсмена и в колонке «тренер» указывает COACH.
  2) В БД athletes.coach_id обновляется.
  3) Тренер в своём кабинете видит агрегированные данные по «своим» athlete_id
     (например, когортный инсайт по журналу достижений — если реализовано в API).

D3. Тренер (COACH)
  1) Входит как coach@…
  2) Видит инструменты тренера (дашборд): то, что API отдаёт для роли COACH —
     обычно список связанных спортсменов и отчёты по ним.
  3) Не редактирует чужие тренировки напрямую, если так не заложено в API —
     смотрите контроллеры (часть операций только у спортсмена или админа).

D4. Администратор (ADMIN)
  1) Входит как admin@…
  2) Управляет пользователями: роль, активность, назначение тренера спортсмену.
  3) Может верифицировать достижения, удалять записи — по правилам эндпоинтов.

D5. Рекомендации (ATHLETE)
  • Система берёт ваши недавние тренировки, тесты, весовую категорию и СТРОКУ
    gold_standards для этой категории (если есть), считает отклонения и план.
  • Кнопка «Сгенерировать» создаёт/обновляет сохранённую рекомендацию в БД.
  • Если эталона для категории нет — логика может показать подсказку или
    использовать запасные значения (см. актуальный код RecommendationService).

D6. Журнал достижений + AI
  • Спортсмен вносит соревнования и недели расписания.
  • Бэкенд агрегирует паттерны и может вызвать LLM для текстового инсайта и
    структурированного «фокуса по дням» — при наличии app.ai.api-key.


================================================================================
E. Техническая связка: как с системой «работают»
================================================================================

1) Фронт (Vue) только отображает данные и вызывает REST API по базовому URL
   /api/... (через VITE_API_BASE_URL).

2) Бэкенд (Spring Boot) проверяет JWT, роль, и отдаёт/принимает JSON или файлы
   (multipart для медиа).

3) БД (PostgreSQL) — источник истины; схема накатывается Liquibase (v1…v6+).

4) Файлы фото/видео — на диске в каталоге app.storage (см. application.yml),
   в БД только метаданные и связь с achievement_id.

5) Почему «в профиле пусто»:
   • Не та роль — блок спортсмена не показывается.
   • ATHLETE без сохранённых полей — только то, что пришло при регистрации.
   • Не тот аккаунт — демо-данные только у athlete@kurash.uz из сида.


================================================================================
F. Роли в коде (кратко)
================================================================================

  role в users: ADMIN | COACH | ATHLETE.

  Регистрация:
    • ATHLETE — /register/athlete (или как в вашем AuthController).
    • COACH — /register/coach.
    • ADMIN — из сида Liquibase или смена роли в админке.

  JWT:
    После логина токен в localStorage (фронт) → заголовок Authorization: Bearer …
    @PreAuthorize("hasRole('ATHLETE')") ограничивает эндпоинты.


================================================================================
G. Ошибка «Athlete profile not found» на странице «Мои тренировки»
================================================================================

Смысл: в токене роль ATHLETE, но в таблице athletes НЕТ строки с вашим user_id.
Типичные причины:
  • Админ в /admin нажал роль ATHLETE у пользователя, у которого не было
    регистрации как спортсмена — роль сменилась, профиль athletes не создался.
  • Редко: сбой при регистрации (пользователь есть, athletes — нет).

Что делать:
  • Войти под демо athlete@kurash.uz — там профиль и сид-тренировки есть.
  • Или снова открыть /admin и ещё раз сохранить роль ATHLETE для этого user:
    бэкенд теперь автоматически создаёт заготовку профиля спортсмена, если её
    не было (регион/вес по умолчанию — поправьте в «Профиле»).
  • На фронте при ошибке загрузки список тренировок очищается, чтобы не показывать
    «чужие» строки из предыдущей сессии.
